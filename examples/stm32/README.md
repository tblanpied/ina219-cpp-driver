# INA219 STM32 Example

This example shows how to use the C++ `ina219` driver on an STM32 target using STM32 HAL generated by **STM32CubeMX**. The application example lives under `App/`, while the low-level MCU initialization, startup, linker, and HAL/CMSIS files come from CubeMX-generated project files.

This project is currently set up for the **STM32U585CIUx** and uses:

- `I2C1` for INA219 communication
- `USART1` for log/output
- CMake as the build system
- ARM GNU Toolchain for cross-compilation
- STM32CubeProgrammer for flashing from the command line

## Project layout

The project is split into two parts:

- `Core/`, `Drivers/`, `cmake/`, startup files, linker script: generated by **STM32CubeMX** for the target MCU and board configuration.
- `App/`: user application code, including the STM32-specific example entry point and any extra C++ code. Keeping the application in `App/` makes regeneration from CubeMX much safer.

Layout:

```text
.
├── App/
│   ├── Inc/
│   └── Src/
│       └── ina219_example.cpp
├── Core/
├── Drivers/
├── cmake/
│   ├── gcc-arm-none-eabi.cmake
│   └── stm32cubemx/
├── CMakeLists.txt
└── STM32CubeMX .ioc file
```

## Requirements

Install the following tools before building:

- **CMake** 3.22 or newer
- **ARM GNU Toolchain** (`arm-none-eabi-gcc`, `arm-none-eabi-g++`, `arm-none-eabi-objcopy`, etc.)
- **STM32CubeMX** if you want to regenerate the STM32 project files
- **STM32CubeProgrammer** if you want to use the `flash` CMake target
- A build tool such as **Ninja** or **GNU Make**

On Linux, make sure the ARM toolchain is available in your `PATH`, and use the provided CMake toolchain file. CMake must use the ARM cross-toolchain rather than the host compiler, otherwise CMSIS/HAL assembly and Cortex-M build steps will fail.

## Build

Configure and build from the repo root:

```bash
cmake -S examples/stm32 -B build
cmake --build build
```

The project uses the ARM GCC toolchain file from `cmake/gcc-arm-none-eabi.cmake`. If needed, you can also pass it explicitly on the command line with `-DCMAKE_TOOLCHAIN_FILE=...`.

## Flash

If `STM32_Programmer_CLI` is installed, the project defines a `flash` target:

```bash
cmake --build build --target flash
```

The flash target uses STM32CubeProgrammer from the command line, typically over **SWD** through an ST-Link probe. The path to `STM32_Programmer_CLI` can be configured from CMake so the project does not depend on a hardcoded local install path.

If CMake cannot find the programmer automatically, configure it manually:

```bash
cmake -S examples/stm32 -B build \
  -DSTM32_PROGRAMMER_CLI=/full/path/to/STM32_Programmer_CLI
```

or:

```bash
cmake -S examples/stm32 -B build \
  -DSTM32_PROGRAMMER_CLI_DIR="$HOME/STMicroelectronics/STM32Cube/STM32CubeProgrammer/bin"
```

## How this example is structured

The INA219 driver itself is kept unchanged. Only the small STM32 platform layer is adapted so the generic driver can use:

- STM32 HAL I2C calls for register access
- `HAL_Delay()` for delays
- `USART1` for logging/output

This is intentional: the driver remains portable, while the platform-specific code stays small and isolated. That is the recommended approach for moving the same application logic across different STM32 devices.

The example code is located in:

```text
App/Src/ina219_example.cpp
```

and is called from the CubeMX-generated `main.c` after peripheral initialization. CubeMX continues to own the generated C startup and init code, while `App/` contains the user-facing C++ example code.

## CubeMX configuration used here

This example was generated from **STM32CubeMX** for:

- **MCU:** `STM32U585CIUx`
- **I2C:** `I2C1`
- **UART:** `USART1`

When regenerating the project from CubeMX, keep the generated files under `Core/`, `Drivers/`, and related CubeMX output directories as generated code, and keep your application logic in `App/`. That separation avoids conflicts during regeneration.

## Porting to another STM32

Porting this example to another STM32 should be straightforward if you keep the same architecture:

1. Create a new CubeMX project for the target STM32.
2. Enable the peripherals you need, typically one I2C instance and one UART instance.
3. Generate the project again as a CMake-based STM32 project.
4. Copy or reuse the `App/` directory.
5. Update only the STM32 platform-specific pieces if peripheral handles or instance names change, for example `hi2c1` or `huart1`.

The goal is to keep:

- `App/` as portable application code
- CubeMX output as the target-specific BSP/HAL layer
- the INA219 driver unchanged

This works best when you treat the firmware as:

- **portable app layer**
- **small platform adapter**
- **CubeMX-generated board/MCU support package**

If you move to a different STM32 family, expect to regenerate:

- clock configuration
- pin assignments
- startup file
- linker script
- interrupt and DMA setup
